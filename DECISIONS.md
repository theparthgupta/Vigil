### 2026-05-23 — Phase 1: synthetic AML dataset (schema + generator)
**What changed:** `data/schema.py` (5 Pydantic models: TransactionRecord, CustomerProfile, Case, plus enums for BusinessType, Direction, Channel, Label, Typology); `data/generator.py` (200 deterministic cases, seed=42); `data/cases_train.json` (160 cases); `data/cases_holdout.json` (40 cases, locked); `data/README.md`.
**Why:** The eval-first principle from the ROADMAP — without a labeled dataset there is nothing to measure, and without a metric there is no basis for any design decision in Phases 2–5.
**What this is:** A seeded Python random generator that produces four typologies of synthetic bank transaction cases (structuring, sanctions hit, rapid passthrough, clean) with Pydantic-validated schemas. Each case carries a ground-truth label and a human-readable notes field explaining the label.
**Why it matters here:** Structuring thresholds are India-specific (CTR = Rs.10 lakh, PMLA s.12). Sanctions names are placeholders for the OpenSanctions /match API (Phase 2 Tool 1). Rapid-passthrough uses the 72h/80% definition consistent with FATF South Asia typology reports. The holdout split is locked so Phase 5 eval numbers are not contaminated by prompt tuning.
**Drawbacks / risks:** 75/25 suspicious-to-clean ratio (3 suspicious typologies, 1 clean) — a naive classifier that always predicts suspicious achieves 75% accuracy. FPR on clean cases must be reported explicitly or the benchmark is misleading. Synthetic data also cannot capture real-world noise (OCR errors, name transliterations, incomplete records).
**Alternatives considered:** 50/50 balance (rejected — would require cutting one suspicious typology or doubling clean cases; 50 clean cases is enough to measure FPR meaningfully); real transaction data (rejected — non-public, PMLA data localisation rules, can't be shared in a public repo); Faker library for names (rejected — adds a dependency for something trivially done with hardcoded lists, and Faker's Indian locale is incomplete).
**How to verify:** `python data/generator.py` exits 0, prints 160 train / 40 holdout, both JSON files pass the structural check script (all required fields, correct counts, labels sane). Re-running produces byte-identical output.

### 2026-05-23 — Phase 2: tool layer (4 investigative tools + 35 tests)
**What changed:** `tools/sanctions.py`, `tools/patterns.py`, `tools/media.py`, `tools/profile.py`; `tests/conftest.py` + 4 test files (35 tests); `pyproject.toml` (pytest + ruff + mypy config); `requirements-dev.txt`.
**Why:** The LangGraph agent (Phase 4) calls these tools to gather evidence. Building and testing them in isolation now means the agent wires together verified components rather than debugging tool logic and orchestration simultaneously.
**What this is:** Four pure Python functions returning plain JSON-serialisable dicts. Tool 1 screens names against sanctions lists (OpenSanctions REST API with local fuzzy-match fallback). Tool 2 extracts AML features deterministically (structuring window, passthrough ratio, velocity, counterparty diversity). Tool 3 searches adverse media via DuckDuckGo Instant Answer API (no key required; stable interface for future swap). Tool 4 structures customer risk facts (business type, account age, prior flags) into a rated summary.
**Why it matters here:** All arithmetic lives in Tool 2 — the LLM Reasoner receives pre-computed features, not raw transactions, so it cannot hallucinate transaction amounts or counts. Tools 1/3/4 are the evidence-gathering layer; the agent's Reasoner synthesises them against RAG-retrieved regulations. Keeping tools as plain dicts (not Pydantic) avoids serialisation friction inside LangGraph state.
**Drawbacks / risks:** Tool 3 (adverse media) uses DuckDuckGo's unofficial instant-answer API — no guaranteed SLA, results are thin for non-public figures. Structuring band uses Rs.8L as lower bound (not Rs.9L) to catch synthetic cases generated from Rs.8.5L; real deployment should recalibrate against actual bank data. Fuzzy-match threshold (0.6) for local sanctions screening may produce false positives on similar-sounding names.
**Alternatives considered:** LLM-based pattern detection for Tool 2 (rejected — violates CLAUDE.md rule: LLM does not do arithmetic); Pydantic return types for tools (rejected — adds coupling; plain dicts flow directly into LangGraph state JSON); Serper API for Tool 3 (rejected — requires paid key; DuckDuckGo is sufficient for a portfolio demo).
**How to verify:** `pytest tests/ -v` → 35 passed. Each suspicious-typology case fires the correct detector; each clean case fires nothing. Sanctions hits found in every sanctions_hit train case; zero hits in clean cases.

### 2026-06-07 — Phase 3: RAG pipeline (section-boundary chunking + Chroma)
**What changed:** `rag/ingest.py` (PDF extraction, section-boundary chunking, OpenAI embedding, Chroma persistence); `rag/retrieve.py` (cosine-similarity retrieval with citation metadata); `tests/test_rag.py` (8 tests); `pypdf` added to `requirements.txt`; `rag/chroma_db/` (gitignored, 893 chunks from 4 documents).
**Why:** The LangGraph Reasoner node needs citable regulatory passages to ground its decisions. Without RAG, the LLM would hallucinate PMLA section numbers and RBI circular references.
**What this is:** A retrieval-augmented generation (RAG) pipeline: PDFs are chunked into semantically coherent sections, embedded with `text-embedding-3-small` (1536-dim), stored in a Chroma vector store, and queried by cosine similarity at inference time. Section-boundary chunking (regex on PMLA/RBI/APG heading patterns) keeps related legal text together rather than breaking mid-clause.
**Why it matters here:** Indian legal documents use inconsistent numbering schemes (PMLA: "12AA."; RBI: "Chapter IV –"; APG: "2.4"). Fixed-size splitting would fragment references like "Section 12AA(1)(b) of the PMLA" across chunks, making citations unretrievable. Metadata (source, section header, page) on every chunk enables the Reasoner to emit precise citations like "RBI KYC MD 2025, para 45, p.46".
**Drawbacks / risks:** The 7-day STR filing timeline (PMLA Maintenance of Records Rules 2005, Rule 3) is NOT in the corpus — those subsidiary rules were not provided as PDFs. The Reasoner will need to note this gap rather than hallucinate the deadline. FIU-IND Reporting Format document capped at page 23 (pages 24+ are pure schema tables that pollute retrieval with field-definition noise). Schema-table filter (>55% short lines) catches residual table chunks.
**Alternatives considered:** LangChain `RecursiveCharacterTextSplitter` (rejected — splits by char count without legal-section awareness; tested and produced many mid-clause breaks); `langchain-community` Chroma integration (rejected — adds a heavy package; raw chromadb + OpenAIEmbeddings is sufficient and more transparent); fixed 512-token chunks with 128-token overlap (rejected — breaks across section numbers, wrecking citation quality).
**How to verify:** `pytest tests/test_rag.py -v` → 8 passed. `python rag/retrieve.py` → Q1 returns RBI KYC Chapter VIII (CTR/STR obligations); Q2 returns RBI para 54 (reporting time limit reference) at rank 3 — exact 7-day text is missing from corpus (data gap); Q3 returns RBI KYC para 45 (PEP definition) at score 0.564.

### 2026-06-07 — Phase 3 addendum: PMLA Rules 2005 ingested + STR-deadline correction
**What changed:** Added 5th document `PMLA_Rules.pdf` (PMLA Maintenance of Records Rules 2005, G.S.R. 444(E)) to the ingest registry → 35 chunks (corpus now 928 total). Fixed a run-on-text chunking bug. Created `agent/prompts.py` with the Reasoner system prompt + STR-deadline guidance. Corrected constraint #1 in ROADMAP.md (two locations).
**Why:** The 7-day STR deadline was the original premise for ingesting this document. On inspection, the document proves the opposite — the current statutory STR deadline is "promptly", not 7 days — a material accuracy issue for an AML product whose generated reports cite legal deadlines.
**What this is:** The PMLA (Maintenance of Records) Rules 2005 are the subsidiary legislation under PMLA 2002 that set reporting deadlines. Rule 8(2), as amended in 2015 (gazette marker `35[...]`), reads: "...on being satisfied that the transaction is suspicious, furnish the information **promptly**...". The original 2005 Rule 8(2) said "not later than seven working days"; the amendment deleted it. The only "seven days" remaining in the document is Rule 9(1D), which governs KYC record updates to CKYCR — not STR filing.
**Why it matters here:** Constraint #1 ("STR filed within 7 days") would have made the Reasoner cite a non-existent statutory deadline — exactly the kind of error a compliance officer catches instantly. Resolution (user-approved Option 2): treat "promptly" as the statutory requirement (cite Rule 8(2) verbatim) and 7 working days as a labelled industry norm / internal SLA. CTR deadline is monthly by the 15th of the succeeding month (Rule 8(1)).
**Drawbacks / risks:** This corpus uses the *current* (amended) Rules; historical cases filed under the pre-2015 7-day regime would need the original text, which is not included. The run-on-chunking heuristic (newline-density < 0.005 → split on inline rule headers) is tuned to this gazette's extraction style and may need adjustment for other run-on PDFs.
**Alternatives considered:** Keep "7 days" as the stated deadline (rejected — factually wrong under current law); drop the 7-day figure entirely (rejected — it is a real, widely-used industry SLA worth surfacing, just not as statute); re-chunk all 5 docs with a unified inline splitter (rejected — the other 4 chunk correctly with line-anchored regex; surgical run-on branch avoids regressions).
**How to verify:** `python rag/ingest.py` (incremental) adds 35 PMLA_Rules chunks. Source-filtered retrieve for Rule 8 returns citation "PMLA (Maintenance of Records) Rules, 2005, 8.Furnishing of information to the Director, p.9" with the verbatim "promptly" text at score 0.62. `agent/prompts.py` STR_DEADLINE_GUIDANCE encodes the statutory-vs-norm distinction.

### 2026-06-07 — Phase 4: LangGraph orchestration (4-node agent + bounded loop)
**What changed:** `agent/state.py` (VigilState TypedDict, 9 fields); `agent/nodes.py` (planner, investigator, reasoner, reporter + route_after_reasoner); `agent/graph.py` (StateGraph wiring); `agent/run_one.py` (E2E runner + LangSmith trace URL); `agent/prompts.py` (+PLANNER_SYSTEM_PROMPT, +REPORTER_SYSTEM_PROMPT); `tests/test_agent.py` (14 tests). No new deps.
**Why:** Phase 4 is the project's core — it composes the dataset, tools, and RAG into an autonomous agent that decides ESCALATE/DISMISS and drafts a citation-grounded STR. This is the LangGraph skill the project exists to demonstrate.
**What this is:** LangGraph models the workflow as a directed graph: START → planner → investigator → reasoner → [conditional] → reporter → END, with a conditional edge looping reasoner→investigator once when confidence < 0.6. State is a TypedDict each node partially updates. LLM nodes (planner, reasoner, reporter) use gpt-4o-mini at temperature=0; the investigator is pure deterministic Python calling the four Phase-2 tools.
**Why it matters here:** Separation of concerns is the design point: the LLM plans/reasons/writes, but ALL arithmetic stays in deterministic tools (CLAUDE.md rule). The reasoner receives pre-computed features, not raw transactions, so it cannot fabricate amounts. Citations come only from RAG passages. The STR_DEADLINE_GUIDANCE flows into both reasoner and reporter, so generated reports correctly state "promptly" (statutory, Rule 8(2)) vs the 7-day norm — verified in the live run.
**Drawbacks / risks:** LLM-based planner adds one call/case and is not perfectly deterministic (mitigated by temp=0). Retrieved passages still include some off-target chunks (e.g. APG Singapore) ranked above the operative ones — the reasoner filters them, but retrieval precision is a Phase-5 tuning target. The confidence threshold (0.6) and the single-loop bound are heuristics not yet validated against metrics.
**Alternatives considered:** Deterministic rule-based planner (rejected by user — LLM planner better demonstrates agentic orchestration); widen-scope vs deeper-retrieval on loop (user chose widen-scope); returning full Pydantic objects through state (rejected — LangGraph state is dict-merged; plain dicts avoid serialisation friction).
**How to verify:** `pytest tests/test_agent.py -v` → 14 passed (router truth table incl. loop bound; investigator evidence assembly on real cases). `pytest tests/ -q` → 57 passed total. `python agent/run_one.py` on case_str_000 → ESCALATE, confidence 0.85, typology=structuring (matches ground truth), 5-section STR citing PMLA Rules Rule 8(2) "promptly" + 7-day-norm caveat, LangSmith trace URL emitted showing all 4 nodes fired (loop correctly did not fire at conf 0.85).

### 2026-06-07 — Phase 5: evaluation harness + baseline→optimized experiment
**What changed:** `eval/metrics.py` (precision/recall/F1/accuracy, FPR-on-clean, per-typology, confusion matrix); `eval/run_eval.py` (concurrent 160-case runner, LangSmith-tagged); `tests/test_metrics.py` (6 tests); `agent/state.py` + `agent/nodes.py` (added `detected_typology` to state); `rag/retrieve.py` (cached Chroma/embedder singletons — concurrency fix); `agent/prompts.py` (the one experimental change); `eval/results_baseline.json`, `eval/results_optimized.json`, `eval/learning_curve.json`.
**Why:** Eval-first is the project's core principle — Phase 5 turns the agent from "it runs" into "it is measured", and produces the baseline-vs-optimized curve the README is built around.
**What this is:** A harness that runs every TRAIN case through the compiled LangGraph agent (concurrently, 8 workers), records decision/confidence/typology/loop/latency, and scores them as a binary suspicious-vs-clean classifier. Each run is tagged in LangSmith ("baseline" / "optimized") for later filtering.
**Why it matters here:** The baseline exposed a precise, single failure: sanctions_hit recall = 0.35 (26 of 40 missed) while every other typology was perfect and FPR-on-clean was 0.00. Tracing one missed case showed the LLM planner was omitting "sanctions" from its plan on domestic-looking cases, so the Reasoner never saw the hit and confidently DISMISSED (conf 0.90).
**The ONE change (experiment):** Planner prompt only — made "profile", "patterns", and "sanctions" MANDATORY in every plan; "adverse_media" stays the single optional tool. No code, retrieval, or threshold change. Result on 160 TRAIN cases: recall 0.783→1.000, F1 0.878→1.000, accuracy 0.838→1.000, sanctions_hit detection 0.35→1.00, with precision and FPR-on-clean unchanged at 1.00 / 0.00 (zero regression).
**Drawbacks / risks:** Scores are PERFECT on the TRAIN split — expected and somewhat unremarkable: the synthetic generator plants clean, separable typology signals that the deterministic tools detect exactly, so once sanctions screening always runs, separation is trivial. This is a tuning-set result, NOT evidence of real-world performance; the locked holdout (and, ultimately, real noisy data) is the honest test. The reliance on an LLM planner means "mandatory" is enforced by prompt, not code — a stronger guarantee would hard-code the baseline tool set. The conditional loop never fired (all confidences ≥0.6), so that path remains unexercised by real data.
**Alternatives considered:** Lower the confidence threshold to force loops on sanctions cases (rejected — treats the symptom, not the missing evidence); hard-code profile+patterns+sanctions in the planner node (viable and more robust, but a code/architecture change, not the single prompt experiment this phase calls for — noted as a future hardening); raise retrieval k (rejected — retrieval wasn't the bottleneck; missing tool evidence was).
**How to verify:** `python eval/run_eval.py --tag baseline` then `--tag optimized` reproduce the two result files; `python eval/metrics.py eval/results_optimized.json` prints F1=1.000, FPR_clean=0.000. `eval/learning_curve.json` holds both metric bundles + deltas. LangSmith shows the two tagged run groups. `pytest tests/ -q` → 63 passed.

### 2026-06-07 — Phase 6: FastAPI backend + Streamlit reviewer UI
**What changed:** `api/main.py` (FastAPI: `GET /health`, `GET /sample`, `POST /investigate`); `app/main.py` (Streamlit reviewer UI); `.claude/launch.json` (preview-server config for the UI).
**Why:** The agent needed a service boundary and a human-in-the-loop surface. The API exposes the compiled graph over HTTP; the UI lets a compliance officer load a case, run the agent, read the STR, and approve/override — the human-in-the-loop framing that is core to the pitch.
**What this is:** FastAPI validates the request body against the Phase-1 `Case` Pydantic model, runs `graph.invoke`, and returns decision/confidence/typology/report/steps/latency. Streamlit is a thin client over those endpoints: it fetches a `/sample` case on startup, renders profile + transactions, and on the Investigate button POSTs to `/investigate` and renders three sections (decision banner, audit trail, STR) plus an approve/override control.
**Why it matters here:** Clean separation — the graph is import-reused by the runner (Phase 4), the eval harness (Phase 5), and now the API, with no duplication. The UI reads cases only via the API's `/sample` (TRAIN split), so the locked holdout is never touched by the frontend. No HTML form tags used (Streamlit-native widgets only).
**Drawbacks / risks:** `/investigate` is synchronous and blocks for the full agent latency (~25–30s incl. the reporter LLM call); fine for a demo, but a production version would stream or queue. No auth/rate-limiting (portfolio demo, no real banking data). The UI uses `use_container_width=`, which Streamlit deprecates after 2025-12-31 — still functional in 1.57 but should migrate to `width=`. The approve/override action is recorded in-session only (not persisted) — a real tool would write an audit record.
**Alternatives considered:** Reuse the `Case` model directly as the request body (chosen — free validation, single source of truth) vs a hand-written request schema (rejected — duplication); Gradio instead of Streamlit (rejected — ROADMAP specifies Streamlit; less control over the reviewer layout); async endpoint (deferred — synchronous is simpler and the demo tolerates the latency).
**How to verify:** `uvicorn api.main:app` then `curl /health` → ok; `curl /sample` → a TRAIN case; `POST /investigate` with that case → ESCALATE + 2.1k-char STR. `streamlit run app/main.py` → UI loads a sample, Investigate renders a red ESCALATE banner (confidence 90%, typology), the STR with citations (e.g. "RBI KYC MD 2025, Section 55, p.54"), the audit trail, and the approve/override toggle — confirmed via live screenshots end-to-end.

### 2026-06-13 — Real-time SSE streaming + Mermaid architecture diagram
**What changed:** Added `POST /investigate/stream` to `api/main.py` — runs the agent via `graph.stream(stream_mode="updates")` and emits Server-Sent Events per node. Rewrote the UI loading logic (`app/static/app.js`, `index.html`, `styles.css`): the 4-stage pipeline + a live status line are now driven by real node events instead of a cosmetic timer. Replaced the README's ASCII architecture diagram with a Mermaid flowchart.
**Why:** The UI froze for ~25s on the synchronous `/investigate` call — a poor demo. Streaming each node as it fires ("Planner selected tools…", "Reasoning against PMLA/RBI…", "Read RBI KYC MD 2025 → ESCALATE (90%)") makes the agent's work visible and is far more compelling. Mermaid renders natively on GitHub for a more polished README.
**What this is:** Server-Sent Events is a one-way HTTP streaming protocol (`text/event-stream`): the server holds the response open and pushes `event:`/`data:` frames. LangGraph's `stream(stream_mode="updates")` yields each node's state delta as it completes; the endpoint maps those deltas to human-readable messages. The browser consumes the stream with `fetch()` + a `ReadableStream` reader (EventSource can't POST a body), parsing frames and updating the pipeline UI live.
**Why it matters here:** Turns a 25s freeze into a narrated investigation that surfaces what the agent actually did — the planned tools, the sanctions-hit count, the regulations read, and the verdict — node by node. Node messages are built from real state deltas, so they can't drift from what happened. The original `/investigate` (non-streaming) is kept for API consumers and the eval harness; streaming is additive.
**Drawbacks / risks:** SSE runs the sync graph generator inside a StreamingResponse, occupying a worker for the request's duration — fine for a demo, but many concurrent investigations would need async/worker scaling. No reconnect/back-pressure handling (a dropped connection just fails the run). Messages are post-node ("done") not intra-node, so a single slow node (the reporter) still shows one "drafting…" status for its whole duration.
**Alternatives considered:** WebSockets (rejected — bidirectional overkill; SSE is simpler for one-way push); `stream_mode="messages"` for token-level LLM streaming (rejected — noisy and per-node summaries read better for this audience); keep the cosmetic timer (rejected — the whole point was to show *real* progress).
**How to verify:** `uvicorn api.main:app` → Investigate; the overlay pipeline lights up node-by-node from real events with a live mono status line, then renders the result on the `done` event.
